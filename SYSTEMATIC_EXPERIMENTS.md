## 详细实验记录

### 1. 基础架构 (Base Architecture)

#### 1.1 LatentLM → StandardGPT

| 项目 | 内容 |
|------|------|
| **假设** | LatentLM 的潜在空间压缩可能有效 |
| **实验** | 同参数量对比 LatentLM vs StandardGPT |
| **结果** | LatentLM 7.66 BPB vs StandardGPT 5.21 BPB |
| **差距** | **-32%** (StandardGPT 大胜) |
| **Insight** | 对于语言建模，标准 Transformer 的直接 token 预测比潜在空间方法更有效。压缩带来的信息损失超过了表示效率的提升。 |
| **行动** | 放弃 LatentLM，以 StandardGPT 为基础 |

#### 1.2 Transformer → Mamba

| 项目 | 内容 |
|------|------|
| **假设** | Mamba 的 O(n) 复杂度可以堆更多层 |
| **实验** | 用纯 PyTorch 实现 Mamba-3 |
| **结果** | Mamba 5.42 BPB vs Transformer 2.49 BPB |
| **差距** | **+118%** (Mamba 惨败) |
| **Insight** | (1) 短序列(128 tokens)下，O(n²) 不是瓶颈，Transformer 的直接 attend 能力更强；(2) 纯 PyTorch 实现太慢，需要 CUDA kernel；(3) Mamba 的优势在长序列(100K+)，不匹配我们的场景 |
| **行动** | 放弃 Mamba，专注 Transformer 优化 |

---

### 2. 激活函数 (Activation Function)

#### 2.1 SwiGLU → LeakyReLU²

| 项目 | 内容 |
|------|------|
| **假设** | 更简单的激活函数可能在小模型上更高效 |
| **实验** | 把 SwiGLU 替换为 `leaky_relu(x, 0.5).square()` |
| **结果** | SwiGLU 2.4939 BPB → LeakyReLU² 2.3875 BPB |
| **差距** | **-4.3%** ✅ |
| **Insight** | (1) SwiGLU 的门控机制在大模型有用，但小模型可能参数不够学好门控；(2) LeakyReLU² 的平方操作提供平滑梯度，负斜率保留负区间信息；(3) 参数效率：LeakyReLU² 没有额外参数，SwiGLU 有门控参数 |
| **行动** | 采用 LeakyReLU²，作为默认激活函数 |

---

### 3. 注意力机制 (Attention Mechanism)

#### 3.1 Full Attention → Sliding Window

| 项目 | 内容 |
|------|------|
| **假设** | 短序列可能不需要全局注意力，局部偏置可能有帮助 |
| **实验** | 改用 window_size=128 的滑动窗口注意力 |
| **结果** | Full 2.4939 BPB → Sliding 2.3568 BPB |
| **差距** | **-5.5%** ✅ |
| **Insight** | (1) 类似 CNN 的局部感受野，为模型提供归纳偏置；(2) 语言的局部依赖通常更强，远距离依赖相对稀疏；(3) 在短序列上，全局注意力可能引入噪声（attend to 不相关的 token）|
| **行动** | 采用 Sliding Window，继续调优 window_size |

### 4. 优化器 (Optimizer)

#### 4.1 AdamW vs Muon

| 项目 | 内容 |
|------|------|
| **假设** | Muon 的 Newton 近似可能加速收敛 |
| **实验** | 固定其他参数，对比 AdamW vs Muon 在不同时长 |
| **结果** | 见下表 |

| 训练时长 | AdamW | Muon | 胜者 |
|----------|-------|------|------|
| 15 min | 4.07 | **3.99** | Muon |
| 30 min | **3.73** | 3.91 | AdamW |
| 60 min | **3.55** | 3.85 | AdamW |

| **差距** | 60min: AdamW 比 Muon 好 **7.8%** |
| **Insight** | (1) Muon 的 Newton 近似在早期给更好的方向，快速下降；(2) 但 Muon 的激进步长在后期可能 overshoot 最优点；(3) AdamW 的保守更新在长期训练更稳定；(4) 论文场景和我们的场景不同，不能盲信 |
| **行动** | 长训练用 AdamW；如需快速验证用 Muon |

### 5. 权重共享 (Weight Sharing)

#### 5.1 独立权重 vs 共享权重（同深度）

| 项目 | 内容 |
|------|------|
| **假设** | 共享权重可以省参数 |
| **实验** | 9 层 Transformer，所有层共享同一组权重 |
| **结果** | 独立 4.08 BPB → 共享 4.15 BPB |
| **差距** | **+1.7%** (更差) |
| **Insight** | 只"舍"没有"取"！省下的参数没有用来增加深度，白白牺牲了表达能力 |
| **行动** | 如果用权重共享，必须同时增加层数 |

#### 5.2 权重共享 + 加深（18层）

| 项目 | 内容 |
|------|------|
| **假设** | 共享权重省下的参数用来加层数，可能更好 |
| **实验** | 18 层共享权重，总参数量接近 9 层独立 |
| **结果** | 18层共享 2.3829 BPB vs 9层独立+优化 2.2821 BPB |
| **差距** | **+4.4%** (仍然更差) |
| **Insight** | (1) 即使加深，共享权重的表达能力仍然有限；(2) 可能需要更多层(24/36)才能看到优势；(3) 在小参数量下，独立权重的多样性更重要 |
| **行动** | 暂时放弃权重共享路线 |

---

### 6. 组合实验 (Combination)

#### 6.1 LeakyReLU² + Sliding Window

| 项目 | 内容 |
|------|------|
| **假设** | 两个有效改进组合应该有累积效果 |
| **实验** | 同时使用 LeakyReLU² 和 Sliding Window |
| **结果** | 单独 LeakyReLU² 2.3875 + 单独 Sliding 2.3568 → 组合 **2.2821** |
| **差距** | 比 baseline **-8.5%**，比单项更好 |
| **Insight** | (1) 两个改进是正交的，可以叠加；(2) 激活函数影响 FFN，注意力影响 Attention，不冲突；(3) 1+1 > 2 的效果说明有协同作用 |
| **行动** | 组合作为新 baseline，继续叠加其他改进 |

## Insight 总结

### 场景匹配原则

> **技术没有绝对好坏，只有是否匹配场景**

| 技术 | 适用场景 | 我们的场景 | 匹配？ |
|------|---------|-----------|--------|
| Mamba | 长序列 (100K+) | 短序列 (128) | ❌ |
| Sliding Window | 短序列 + 局部依赖 | 短序列 (128) | ✅ |
| Muon | 短训练 (<30min) | 长训练 (60min+) | ❌ |
| QAT | 需要量化 | 需要 3-bit | ✅ (待试) |

### Trade-off 利用原则

> **Trade-off 只有在"取"被利用时才有意义**

| 错误做法 | 正确做法 |
|----------|---------|
| 权重共享但不加层 | 权重共享 + 加倍层数 |
| 用 Diffusion 但评估 BPP | 用 Diffusion 做并行生成任务 |
| 用 Mamba 处理短序列 | 用 Mamba 处理 100K+ 序列 |

### 简单优先原则

> **在小模型上，简单方法往往更有效**

| 复杂方法 | 简单方法 | 结果 |
|----------|---------|------|
| SwiGLU (门控) | LeakyReLU² (平方) | 简单赢 |
| Muon (Newton 近似) | AdamW (动量+自适应) | 简单赢 |
| 全局注意力 | 滑动窗口 | 简单赢 |
